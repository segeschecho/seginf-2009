La segunda parte del trabajo consistió en la realización de reportes sobre el tráfico capturado por el sniffer.
Nosotros construimos una herramienta que denominamos reporTool. La misma permite realizar los distintos análisis de tráfico (reportes) generando una salida en formato pdf o en formato html mediante una interfaz gráfica.

\section{Reportes}
Los siguientes son los reportes incluidos con reporTool

\subsection{Reporte de horario laboral}
Este reporte permite analizar el tráfico discriminandolo según un periodo laboral (días de la semana, y rango horario). De esta manera el reporte puede mostrar que usuarios usaron Internet fuera de dichos horarios y que porcentaje de su actividad estuvo en infracción. El reporte puede mostrar todos los pedidos que se realizaron fuera de hora o solo mostrar el porcentaje en infracción.

Ademas puede mostrar la distribución del uso de Internet a lo largo de las distintas horas del día.

\subsection{Reporte por listas negras (blacklists)}
Este reporte permite especificar en un archivo una lista negra con dominios. El reporte analiza el tráfico buscando visitas a alguno de esos dominios. Es en realidad una familia de reportes, ya que es posible utilizar varias instancias del mismo utilizando distintas listas negras, con distintos tipos de dominios, por ejemplo sitios de apuestas, de warez, etc. 

Si bien reporTool trae unas listas negras para ciertas categorías, es fácil modificar esas listas, o reemplazarlas por otras, crear un reporte nuevo que trabaje con listas de otras categorías.

\subsection{Reporte por contenido (content-type)}
En este reporte podemos ver el tipo de contenido del tráfico, básicamente hicimos seis separaciones distintas (audio, video, aplicación, multipart, imagen y texto). En el reporte podemos ver, por ejemplo, la cantidad de tráfico de tipo imagen y la cantidad de tráfico que no es imagen. También tenemos la cantidad de tráfico de imagen separada por usuario para poder ver más en detalle qué hace cada usuario.

Si bien se podrían hacer varias separaciones mas (como por ejemplo discriminar por tráfico de tipo jpeg o pdf) estas separaciones generales nos parecieron las más interesantes para mostrar en el reporte.
Igualmente es factible hacer algunas de estas consultas mediante el modo interactivo (o extendiendo el reporte).

\subsection{Reporte por tráfico de nivel de aplicación}
En este reporte se retoma el análisis del tráfico según protocolo de la capa de aplicación (ya al guardar los requests en la base de datos, los distinguimos entre tráfico http y no http).

Aprovechando esta división, analizamos por un lado cada tipo de tráfico. Del tráfico http se analiza únicamente el volumen y los IPs involucrados (para poder hacer el reporte de tráfico discriminado por usuario, y el de tráfico total). Por otro lado, buscamos 3 protocolos distintos en el tráfico no http: SSL, TLS (el sucesor de SSL) y SSH.

La forma de distinguir paquetes SSL y TLS es esencialmente la misma: se revisa el segundo y tercer byte de cada paquete, buscando una serie de números (los distintos números de versión del protocolo). Si el protocolo es SSL tendrá un cierto número de versión, si es TLS otro (hay mas de un número de versión de TLS).

\begin{figure}[H]
\centering
\includegraphics[scale=0.4]{./figuras/TLS.png}
\caption{Un paquete SSL / TLS}
\end{figure}

En cambio, para distinguir si se trata de un paquete SSH, hay que buscar el string "SSH-versiónProtocolo-versiónSoftware", que está en texto en claro. Además de este string podemos obtener el software utilizado. Uno de los reportes muestra los clientes SSH utilizados: nuestra suposición es que dada la topología de la red (habiendo que pasar a través de un proxy http) se puede hacer SSH hacia afuera de la red y no hacia adentro. Entonces, analizamos que software se utilizó mirando los request no http (y no lo responses) y obtenemos cual es el software que está permitiendo al infractor hacer SSH hacia afuera de la red.

\subsection{Reporte de seguimiento}
Este reporte permite elegir hasta 5 sitios y analizar la evolución de la cantidad de visitas y de tráfico mes a mes para estos sitios, a lo largo de un periodo de tiempo determinado.

\subsection{Reporte por heurística}
Este reporte puede pensarse de alguna forma como un complemento para el reporte por blacklists. El mismo utiliza un archivo donde se definen ciertas palabras clave (por ejemplo, palabras sobre apuestas: Póker, Black Jack, casino, etc) que son buscadas en las requests y en los responses. El reporte muestra luego para que usuarios se encontraron matches, así como cuales fueron los principales matches.

Las listas negras solo encuentran requests a los dominios definidos en ellos, entonces es factible que un usuario pueda visitar alguna página que no está en la lista pero que debería estarlo, o que incluso saltee el chequeo de las listas negras (por ejemplo navegando con el cache de Google). La búsqueda heurística permite detectar estas cosas, al costo de posiblemente dar falsos positivos sobre el contenido de las visitas. Sin embargo puede ser una herramienta útil para detectar usos indebidos.

\section{Agregado de nuevos reportes}
ReporTool fue pensado para ser fácilmente extensible, es decir que sea fácil poder agregar nuevos reportes definidos por el usuario.

Para poder definir un nuevo reporte, es necesario definir una clase que herede de Reporte. Esta misma debe implementar el método ejecutar que recibe un desde y un hasta, que son las fechas entre las cuales debe analizarse el tráfico.

Dicho método debe devolver un string latex, que es el que se utilizará para generar el pdf o el html correspondiente al reporte.

Una vez definida esta clase, debe importarse en reporTool, crear una instancia, envolverla con un Configurador y agregar al configurador a la lista de configuradores.
 
Para ilustrar un poco mejor esto, presentaremos un ejemplo muy sencillo de como se define y se agrega un nuevo reporte.

\section{Algunas consideraciones de seguridad sobre ReporTool}
ReporTool es una herramienta pensada para ser usada por un usuario administrador, ya que la misma tiene acceso a la base de datos, y si la pudiera usar cualquier usuario, podría modificarla o borrarla a su antojo.

Otra consideración que hay que hacer, es que al identificar a los usuarios por su IP un usuario malicioso podría inculpar a otro usuario inocente. Para esto no tendría mas que generar requests a sitios prohibidos por la blacklist spoofeando la IP de la víctima. Entonces al correr reporTool, en el reporte aparecerá que la víctima estuvo accediendo a sitios indebidos, cuando en verdad eso es falso.
